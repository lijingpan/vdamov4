# 商品属性与规格设计

## 1. 设计目标

当前新系统的商品模型只有基础字段：

- 商品名
- 商品编码
- 分类编码
- 售价
- 上下架状态

这只能支撑“单价单规格商品列表”，无法支撑老系统里已经存在的以下场景：

- 单规格 / 多规格商品
- 称重商品
- 商品属性
- 加料商品
- SKU 级价格、库存、条码、编码
- 后续点单、加菜、库存扣减、报表按 SKU 统计

因此，商品管理不建议只“再加一个普通属性字段”了，而是要把商品升级成一个聚合模型：

- 商品主档 `SPU`
- 商品规格定义
- 商品 SKU
- 商品属性
- 商品加料关系

一期建议先把“商品属性 + 商品规格 + SKU”做完整闭环；套餐、复杂加料模板、时段售卖先留扩展位。

## 2. 结论

### 2.1 是否要新增商品属性

要，但不是在 `da_product` 上简单补一个 `attribute` 文本字段。

建议新增两层能力：

1. 商品主档上的业务属性开关
2. 商品属性值明细表

主档层面至少补以下字段：

- `product_type`：商品类型，`NORMAL` / `WEIGHED` / `ADD_ON` / `SET_MEAL`
- `spec_mode`：规格模式，`SINGLE` / `MULTI`
- `attr_enabled`：是否启用商品属性
- `material_enabled`：是否启用加料
- `weighed_enabled`：是否称重商品

其中真正回答“是不是还要新增一个属性”的最小答案是：

- 要新增 `product_type`

原因：

- 它能区分标准商品、称重商品、加料商品、套餐商品
- 后续点单和收银逻辑会直接依赖它
- 单靠 `weighed_enabled` 或 `material_enabled` 这种布尔值，无法表达商品的主业务类型

### 2.2 商品规格怎么做

规格不能直接塞在商品表里，应该拆为：

- 规格模板
- 规格值
- 商品绑定的规格组合
- SKU 明细

规格的核心目标不是“显示一个多规格表格”，而是让每一个可售卖组合都有独立 SKU。

### 2.3 商品属性怎么做

商品属性和规格不是一回事。

建议区分：

- 规格：影响 SKU 组合和价格库存，例如“大杯/中杯”“冷热”
- 属性：展示或点单辅助信息，不生成 SKU，例如“辣度”“推荐标签”“默认口味”

老系统里的“属性值可设置默认值”是合理的，新系统保留。

## 3. 建议的数据模型

基于当前项目的表规范，建议新增以下表。

### 3.1 调整主表 `da_product`

保留现有字段，并补充：

- `product_type` varchar(32) not null
- `spec_mode` varchar(16) not null default 'SINGLE'
- `image_url` varchar(255) null
- `description` varchar(500) null
- `sort_order` int not null default 0
- `weighed_enabled` boolean not null default false
- `attr_enabled` boolean not null default false
- `material_enabled` boolean not null default false
- `default_sku_id` bigint null

说明：

- `price_in_cent` 一期可以先保留，作为单规格商品默认售价或列表冗余价格
- 真正成交价格以后以 `da_product_sku.price_in_cent` 为准

### 3.2 商品 SKU 表 `da_product_sku`

每个商品至少有一条 SKU，单规格商品也是一条。

建议字段：

- `id`
- `product_id`
- `store_id`
- `sku_code`
- `barcode`
- `spec_key`
- `spec_name`
- `price_in_cent`
- `line_price_in_cent`
- `cost_price_in_cent`
- `box_fee_in_cent`
- `stock_qty`
- `auto_replenish`
- `weight_unit_gram`
- `sort_order`
- `active`

字段含义：

- `spec_key`：规格组合唯一键，例如 `SIZE=L|TEMP=HOT`
- `spec_name`：展示名，例如 `大杯/热`
- `weight_unit_gram`：称重基准克数，例如老系统里的 `component`

说明：

- 单规格商品也走 SKU，前端不用分两套提交结构
- 称重商品仍然落在 SKU 上，因为不同规格可能对应不同称重单价

### 3.3 规格模板主表 `da_product_spec`

建议字段：

- `id`
- `store_id`
- `name`
- `sort_order`
- `image_enabled`
- `active`

说明：

- 对应老系统里的“规格”
- 一期先按门店维度维护即可；后面如果走总部下发，再补模板来源字段

### 3.4 规格值表 `da_product_spec_value`

建议字段：

- `id`
- `spec_id`
- `store_id`
- `name`
- `image_url`
- `sort_order`
- `active`

### 3.5 商品规格绑定表 `da_product_spec_rel`

表示一个商品用了哪些规格、显示顺序如何。

建议字段：

- `id`
- `product_id`
- `store_id`
- `spec_id`
- `sort_order`

### 3.6 商品属性主表 `da_product_attr`

建议字段：

- `id`
- `store_id`
- `name`
- `sort_order`
- `active`

说明：

- 对应老系统里的“商品属性”
- 一条属性下面有多个属性值

### 3.7 商品属性值表 `da_product_attr_value`

建议字段：

- `id`
- `attr_id`
- `store_id`
- `name`
- `sort_order`
- `active`

### 3.8 商品属性绑定表 `da_product_attr_rel`

表示商品选择了哪些属性、哪些值被选中、默认值是什么。

建议字段：

- `id`
- `product_id`
- `store_id`
- `attr_id`
- `attr_value_id`
- `is_default`
- `sort_order`

约束建议：

- 同一商品同一属性下可以多选
- 同一商品同一属性最多一个默认值

### 3.9 商品加料绑定表 `da_product_material_rel`

一期不先做复杂加料组，先预留简单关联。

建议字段：

- `id`
- `product_id`
- `store_id`
- `material_product_id`
- `sort_order`
- `default_selected`

说明：

- 加料商品本身也属于 `da_product`
- 当 `product_type = ADD_ON` 时，可作为别的商品的加料项

## 4. 建议的领域规则

### 4.1 商品类型规则

- `NORMAL`：普通商品，可单规格或多规格
- `WEIGHED`：称重商品，必须至少一个 SKU，SKU 必须有 `weight_unit_gram`
- `ADD_ON`：加料商品，默认不直接出现在普通商品列表，可被其他商品引用
- `SET_MEAL`：套餐商品，一期先占位，不做套餐子项编排

### 4.2 规格规则

- `spec_mode = SINGLE` 时，系统自动维护一条默认 SKU
- `spec_mode = MULTI` 时，至少绑定一个规格，SKU 组合由规格值笛卡尔积生成
- 同一商品下 `spec_key` 必须唯一
- 停用规格值后，历史 SKU 不物理删除，只允许失效

### 4.3 属性规则

- 属性只做展示/点单选择，不参与 SKU 唯一键
- 属性值可多选
- 每个属性最多设置一个默认值
- 如果某属性全部未勾选，则视为未配置默认值

### 4.4 称重规则

- 称重商品必须绑定 SKU
- `price_in_cent` 表示该 SKU 对应 `weight_unit_gram` 的价格
- 收银端按 `实际称重克数 / weight_unit_gram * price_in_cent` 计算
- `weight_unit_gram` 建议限制为正整数，如 `100`、`500`

### 4.5 库存规则

- 一期库存先做到 SKU 级当前库存字段
- 不单独做库存流水表时，只允许后台维护期初库存
- 订单扣减时直接扣 SKU 库存，为后续库存流水扩展预留接口

## 6. 前端页面设计建议

## 6.1 商品列表页

商品列表建议新增列：

- 商品类型
- 规格模式
- 基础售价 / 价格区间
- SKU 数
- 属性开关
- 状态

列表展示规则：

- 单规格显示单价
- 多规格显示 `最低价 ~ 最高价`
- 称重商品增加“称重”标签

## 6.2 商品编辑页

建议拆成 4 个区域，和老系统接近，但结构更清晰。

### 区块一：基础信息

- 门店
- 商品名称
- 商品编码
- 商品分类
- 商品类型
- 商品图片
- 商品描述
- 上下架状态

### 区块二：价格 / 规格信息

- 称重商品开关
- 商品规格模式：单规格 / 多规格
- 规格选择器
- SKU 表格

SKU 表格列建议：

- 规格组合
- 销售价
- 划线价
- 成本价
- 包装费
- 初始库存
- 商品重量基准
- 商品条码
- 商品编码

这部分基本沿用老系统交互，但新系统建议去掉“页面上临时计算笛卡尔积后直接拼裸对象”的写法，改成前端明确维护：

- `selectedSpecs`
- `skuRows`

### 区块三：商品属性

- 属性开关
- 选择属性模板
- 勾选属性值
- 指定默认值

交互规则：

- 属性值可多选
- 默认值单选
- 未启用时整个区块折叠

### 区块四：加料商品

- 加料开关
- 选择可关联的加料商品

一期先做简单关联，不做：

- 加料分组
- 最少/最多可选
- 必选/可选
- 加料阶梯价

## 7. 一期落地边界

为了避免商品模块一次性做得过重，一期建议只交付下面这些：

### 必做

- 商品类型 `product_type`
- 规格模式 `spec_mode`
- 商品 SKU 表
- 规格模板 + 规格值
- 商品属性模板 + 属性值
- 商品编辑页支持单规格 / 多规格
- 商品编辑页支持属性勾选和默认值
- 列表页显示价格区间、规格模式、商品类型

### 可选

- 称重商品 `weight_unit_gram`

如果收银端近期要接，就一期一起上；否则字段先落表、前端先隐藏。

### 暂不做

- 套餐子项编排
- 多语言商品翻译表
- 门店商品覆盖价
- 时段售卖
- 限购
- 复杂加料组
- 规格图片
- 库存流水

## 9. 推荐实施顺序

1. 先补库表和实体，完成商品主档 + SKU 聚合结构。
2. 再做规格模板和属性模板管理页。
3. 最后升级商品编辑页，接入多规格和属性。

这样拆的原因：

- 商品编辑页依赖模板数据
- 订单和收银后续只需要消费 SKU，不需要关心前端的选择过程
- 先把 `SPU + SKU` 立住，后续加料、套餐、门店覆盖价都能继续长

## 10. 最小决策建议

如果现在只做最小决策，我建议立即确认以下三件事：

1. `da_product` 新增 `product_type`，不要只加普通文本属性字段。
2. 商品规格按 `da_product_sku` 独立建模，单规格也走 SKU。
3. 商品属性与规格分离，属性不参与 SKU 唯一键。

这三个点一旦确定，后面的表结构和页面设计就不会反复推倒。
