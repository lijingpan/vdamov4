# 点餐系统技术方案

## 1. 方案目标

本方案用于支撑点餐系统一期建设，重点解决以下问题：

- 支撑门店、桌位、商品、会员、订单、优惠、设备等后台管理能力
- 支撑订单、加菜、结账等核心业务服务能力
- 支撑后台端桌台状态、订单状态的实时刷新
- 支撑后台端与菜单数据的三语言国际化
- 在高峰期保证订单、加菜、结账的稳定性和性能
- 为二期收银 App、离线能力、扫码点餐、厨房联动等扩展预留空间

## 2. 技术选型结论

本项目后端技术路线确定为 Java。

### 2.1 一期前端选型

推荐方案：

- 管理后台：Vue 3 + Vite + TypeScript + Element Plus
- 状态管理：Pinia
- 路由：Vue Router
- 网络请求：Axios
- 实时通信：WebSocket
- 国际化：Vue I18n

说明：

- 如果前端使用 Vue 3，则不应再使用老的 Element UI，而应使用 Element Plus。
- Element UI 主要对应 Vue 2 生态，新项目直接上 Vue 3 时，组件库应统一为 Element Plus。
- 一期前端只建设后台管理端，不建设顾客点餐前台和收银 App 页面。
- 后台管理端适合使用 Element Plus，组件成熟、表格和表单能力完整，适合商品、订单、门店、桌位、会员等后台页面。
- 一期后台端按简体中文、英文、泰文三语言实现。

### 2.2 一期后端选型

推荐方案：

- 后端语言：Java 21
- 后端框架：Spring Boot 3.x
- Web 框架：Spring MVC 为主，实时推送模块使用 WebSocket
- 数据库：MySQL 8.0
- 缓存与消息：Redis
- ORM：MyBatis-Plus
- 网关：Nginx
- 对象存储：MinIO 
- 日志与监控：Spring Boot Actuator + Prometheus + Grafana
- 国际化：Spring MessageSource + 数据字典多语言方案

结论说明：

- 这个项目的性能瓶颈不在普通后台 CRUD，而在桌台状态、订单状态、结账状态的高并发刷新。
- 这类场景的关键不是单纯换更“快”的语言，而是避免全量轮询。
- 对于门店管理系统、收银系统、点餐后台这类偏事务型系统，Java + Spring Boot 的稳定性、事务能力、团队接手成本、生态完整度更合适。
- 真正要解决“桌台一直刷新”的问题，应采用“事件推送”而不是“前端每 2 秒轮询一次”的做法。
- 国际化不要只做页面文案，菜单、商品、加料、优惠等业务数据也要支持多语言。

## 3. 为什么推荐 Spring Boot

推荐 Spring Boot 作为主后端，而不是优先选 Node.js 或 Go，原因如下：

- 订单、结账、优惠、会员余额、积分抵扣本质上都是强事务场景
- Spring Boot 在事务控制、权限体系、后台管理、日志监控、数据库生态上更成熟
- WebSocket、Redis、MySQL 的集成方案成熟，适合做门店系统
- 后续如果要接打印、小票、会员中心、营销规则、报表，Java 方案更稳

不优先推荐的方案：

- Node.js：开发速度快，但在复杂事务和大型后台治理上通常不如 Java 稳
- Go：并发能力强，但业务后台生态、权限、ORM、团队上手成本通常不如 Spring Boot 平衡

因此建议：

- 主业务后端统一采用 Spring Boot
- 用架构设计解决实时性能问题，而不是把希望寄托在语言切换上

## 4. 总体架构

建议采用分层架构：

1. 前端层
2. 网关层
3. 应用服务层
4. 缓存与消息层
5. 数据存储层

建议系统结构如下：

- 管理后台：商品、门店、桌位、设备、订单、会员、优惠配置
- 业务服务端：统一业务 API、权限、订单事务、优惠计算
- Redis：缓存热点数据、做消息广播、控制部分并发
- MySQL：存储订单、商品、桌位、会员、优惠等核心数据
- WebSocket 服务：向管理后台推送桌位与订单变化

二期预留：

- 收银 App：开台、点餐、加菜、结账、桌台管理
- 顾客点餐前台：扫码点餐、购物车、订单确认
- 离线同步：本地缓存、补偿同步、冲突处理

国际化要求：

- 后台界面文案支持简体中文、英文、泰文
- 菜单和商品业务数据支持简体中文、英文、泰文
- 门店可配置默认展示语言
- 会员资料需支持国际化手机号

## 5. 实时刷新方案

### 5.1 不推荐的做法

不建议后台端通过短轮询持续刷新桌台列表，例如：

- 每 2 秒调用一次桌台列表接口
- 每 2 秒调用一次订单列表接口
- 每个后台客户端都在做全量刷新

问题：

- 门店越多、桌位越多，请求量越大
- 大量无效查询会打满数据库
- 高峰期结账时会和正常事务抢资源
- 前端界面会频繁重绘，体验也不好

### 5.2 推荐的做法

推荐使用“初始化拉取 + 增量推送 + 断线补偿轮询”：

#### A. 初始化拉取

- 页面首次进入时，调用一次桌台列表接口
- 拉取当前门店的桌位状态、当前订单摘要、金额摘要

#### B. 增量推送

当以下事件发生时：

- 开台
- 下单
- 加菜
- 退菜
- 结账
- 清台
- 桌位状态变更

后端生成事件消息，并通过 WebSocket 推送给相关门店在线后台客户端。

#### C. 断线补偿

- WebSocket 断开后自动重连
- 重连成功后重新拉取当前门店桌台快照
- 兜底再加低频轮询，例如 30 秒或 60 秒一次

这样可以把高频全量查询改成低频快照 + 实时增量更新。

### 5.3 推荐事件流

建议实时事件流如下：

1. 用户执行下单、加菜、结账
2. 后端事务提交成功
3. 后端写入 MySQL
4. 后端更新 Redis 缓存
5. 后端发布桌台变更事件到 Redis Pub/Sub
6. WebSocket 节点订阅消息并推送给对应门店后台客户端
7. 前端只更新变更的那张桌台或订单卡片

## 6. 后端模块划分

建议一期先按模块化单体建设，不要一开始就拆成很多微服务。

推荐模块：

- 用户与权限模块
- 门店模块
- 桌位模块
- 设备模块
- 商品模块
- 订单模块
- 优惠模块
- 会员模块
- 支付与结账模块
- 实时推送模块
- 报表模块
- 国际化模块

二期扩展模块预留：

- 收银作业模块
- 离线同步模块
- 点餐前台模块

原因：

- 当前一期业务复杂度还不足以证明必须上微服务
- 微服务会显著增加部署、联调、排障和事务处理成本
- 先做模块化单体，更适合快速落地
- 后续如果门店规模和并发量明显增长，再按模块拆服务

## 7. 数据库设计建议

核心库建议使用 MySQL 8.0。

核心表建议包括：

- `store` 门店表
- `table_info` 桌位表
- `table_area` 桌位区域表
- `store_device` 门店设备表
- `store_device_rule` 门店设备规则表
- `device_log` 设备日志表
- `product` 商品表
- `product_category` 商品分类表
- `product_option_group` 加料组表
- `product_option_item` 加料项表
- `i18n_message` 系统文案多语言表
- `product_i18n` 商品多语言表
- `product_category_i18n` 商品分类多语言表
- `product_option_group_i18n` 加料组多语言表
- `product_option_item_i18n` 加料项多语言表
- `order_main` 订单主表
- `order_item` 订单明细表
- `order_append_log` 加菜记录表
- `order_discount` 订单优惠明细表
- `payment_record` 支付记录表
- `member` 会员表
- `coupon` 优惠券表
- `sync_log` 同步日志表
- `device_operation_log` 设备操作日志表

设计原则：

- 金额字段统一使用分为单位的整数，避免浮点误差
- 订单号、支付单号、退款单号必须全局唯一
- 订单状态流转要有操作日志
- 加菜记录建议独立保留，便于审计和报表统计
- 为二期离线同步预留 `version`、`updated_at`、设备标识字段
- 多语言数据不要硬编码在前端，统一由数据库维护
- 业务表保留默认语言字段，多语言内容单独建扩展表
- 设备配置和测试记录独立留痕，便于排障
- 会员手机号按国际号码标准设计，不按单一国家规则写死

推荐多语言数据模型：

- 基础表保留默认语言内容，例如中文名称
- 多语言扩展表按 `biz_type + biz_id + lang_code` 存储翻译内容
- `lang_code` 建议统一使用 `zh-CN`、`en-US`、`th-TH`
- 查询时优先取当前语言，缺失时回退默认语言

会员国际化数据建议：

- 会员表拆分 `country_code`、`phone_national`、`phone_e164`
- `phone_e164` 作为统一检索和唯一约束主字段
- 至少预置中国 `+86`、泰国 `+66`
- 前端录入时由区号和本地号码拼装，后端统一归一化
- 会员搜索时支持按完整国际号和本地号模糊检索

## 8. Redis 使用建议

Redis 主要承担三类职责：

### 8.1 缓存热点数据

- 门店基础信息
- 桌位状态摘要
- 商品列表缓存
- 门店可售商品缓存
- 优惠规则缓存

### 8.2 实时消息分发

- 门店桌位变化通知
- 订单变化通知
- 结账状态变化通知
- 设备状态变化通知

### 8.3 并发控制

- 订单提交幂等控制
- 结账请求幂等控制
- 库存扣减或优惠券核销的原子控制

建议：

- 简单广播用 Redis Pub/Sub
- 需要更强可追溯能力时可升级到 Redis Streams 或 MQ
- 高并发扣减场景可用 Lua 脚本保证原子性

适用页面：

二期收银 App 先不在一期实现，但架构上需要预留：

- 设备身份标识
- 收银员登录与门店绑定
- 本地订单草稿能力
- 本地缓存商品、桌位、优惠、会员摘要数据
- 离线单据同步机制
- 冲突处理和补偿机制

建议一期后端预留以下能力：

- 统一订单号规则
- 客户端提交幂等号
- 数据版本号或更新时间字段
- 同步日志表
- 设备维度操作日志

### 9.3 前端性能建议

- 桌台列表和订单列表采用增量更新，避免每次全量重绘
- 大列表视图使用虚拟滚动
- 将门店、桌位、订单摘要放入 Pinia，统一维护状态
- WebSocket 消息只传最小变更字段，不传整页数据

## 10. 接口设计建议

接口风格建议使用 REST API + WebSocket 事件推送。

国际化规则：

- 请求头支持 `Accept-Language`
- 用户维度保存当前语言设置
- 接口响应中的状态描述、异常消息按语言返回
- 菜单相关接口按指定语言返回翻译内容
- 设备相关接口按指定语言返回类型、状态、规则说明
- 会员接口按统一国际手机号格式返回

### 10.1 REST API

适合：

- 商品查询
- 下单
- 加菜提交
- 结账提交
- 优惠试算
- 订单查询
- 门店与桌位配置
- 多语言菜单维护
- 设备管理与规则维护
- 打印测试与播报测试
- 国际会员查询与绑定

### 10.2 WebSocket

适合：

- 桌位状态变更通知
- 新订单通知
- 加菜通知
- 结账完成通知
- 后台订单列表实时刷新

建议按门店订阅频道：

- `/topic/store/{storeId}/tables`
- `/topic/store/{storeId}/orders`

## 11. 高并发与稳定性设计

### 11.1 幂等控制

以下接口必须幂等：

- 提交订单
- 加菜提交
- 结账支付回调
- 优惠券核销

实现建议：

- 前端提交唯一请求号
- 后端用 Redis 或数据库唯一索引防重

### 11.2 事务控制

以下操作必须放在事务内：

- 创建订单 + 写订单明细
- 加菜 + 更新订单金额
- 结账 + 写支付记录 + 更新订单状态
- 优惠券核销 + 订单完成

### 11.3 缓存一致性

建议模式：

- 先写数据库
- 事务成功后删除或更新缓存
- 再发送实时事件

不要只改缓存不落库。

### 11.4 表状态刷新性能

重点优化点：

- 桌台页只看当前门店，不查全量门店
- 桌台页返回摘要字段，不返回大对象
- 每次变更只推送受影响桌台
- 定时全量同步作为兜底，不作为主刷新机制

### 11.5 设备接入建议

一期设备管理先以“后台配置 + 状态展示 + 规则维护 + 测试能力”为主，不在一期深度实现本地驱动控制。

建议支持的设备类型：

- 云小票打印机
- 后厨打印机
- 标签打印机
- 云语音音响

建议实现方式：

- 后台维护设备基础信息、启停状态、用途、规则
- 通过厂商开放接口或中间服务完成打印测试、播报测试
- 后端记录测试请求、返回结果、失败原因
- 设备异常状态通过轮询厂商接口或回调方式同步

打印机规则建议：

- 按业务类型区分打印，如堂内、外送、自取
- 按场景区分打印，如下单、加菜、结账
- 支持一个门店多打印机并行配置
- 支持打印模板与打印规则分离

音响规则建议：

- 按场景触发播报，如新订单、加菜、结账完成
- 支持门店级播报开关
- 支持播报内容模板配置
- 支持测试播报

设备日志建议：

- 设备新增、编辑、启停
- 打印测试、播报测试
- 实际打印请求、播报请求
- 失败原因与失败次数

### 11.6 二期离线能力预留

二期如果要做收银 App 离线，建议采用“本地可操作、联网后补同步”的模式，但一期不实现完整离线闭环，只做后端预留。

一期需要预留：

- 所有核心写接口支持幂等提交
- 订单、支付、加菜、退款都要有唯一业务流水号
- 关键表保留 `version` 或 `updated_at`
- 提供按时间游标或流水号的增量同步接口
- 设备操作日志可回放、可追踪

二期实现离线时，再补：

- App 本地数据库
- 同步队列
- 冲突解决策略
- 离线支付边界

### 11.7 国际化实现建议

后台文案国际化建议：

- 前端使用 `Vue I18n` 管理静态文案
- 后端使用 `MessageSource` 管理错误消息、系统提示、状态文案
- 语言切换支持用户维度保存

菜单数据国际化建议：

- 商品、分类、加料、优惠等业务数据使用多语言扩展表
- 接口按语言返回目标内容，避免前端自己拼接翻译
- 管理后台提供三语言录入页签
- 若缺少翻译内容，按默认语言回退

会员国际化建议：

- 手机号统一存储为国际格式
- 会员注册、查询、绑定、下单都以国际手机号为主键入口之一
- 后端不要使用仅适配中国大陆手机号的正则
- 国家区号字典独立维护，便于后续继续扩展其他国家

报表国际化建议：

- 页面内报表标题、筛选项、状态说明支持三语言
- 导出文件名和表头支持按当前语言输出

## 12. 部署建议

一期建议部署方式：

- 1 台 Nginx
- 2 台 Spring Boot 应用实例
- 1 套 MySQL 主库
- 1 套 Redis

这样可以先满足小到中等规模门店系统。

后续扩展方向：

- 应用实例横向扩容
- MySQL 主从或读写分离
- Redis 哨兵或集群
- 独立消息队列
- 独立报表库

## 13. 开发顺序建议

1. 搭建基础框架与权限体系
2. 完成门店、桌位、商品、加料、优惠、会员基础资料模块
3. 完成门店设备管理能力
4. 完成订单、加菜、结账的后台能力
5. 完成后台端 WebSocket 实时刷新
6. 完成后台端国际化和菜单多语言能力
7. 完成报表、日志与监控
8. 二期再建设收银 App 和离线同步能力

## 14. 风险与注意事项

- 如果前端坚持使用 Element UI，会和 Vue 3 生态产生兼容问题
- 如果后台桌台页采用高频轮询，后期门店一多性能一定出问题
- 如果订单、支付、优惠没有幂等和事务设计，金额问题会很多
- 如果一开始就上微服务，项目复杂度会被显著放大
- 如果二期要做离线收银，但一期没有预留同步标识和幂等能力，后面改造成本会很高
- 如果菜单多语言直接写死在前端，后续维护和翻译发布会很重
- 如果设备管理不做日志留痕，后续打印和播报问题会很难排查

## 15. 最终建议

最终建议如下：

- 前端：Vue 3 + Vite + TypeScript + Element Plus
- 后端：Java 21 + Spring Boot 3.x + MySQL 8.0 + Redis
- 一期范围：后台管理端 + 业务服务端
- 国际化：简体中文 + 英文 + 泰文
- 会员：支持国际手机号，首批覆盖中国 `+86` 与泰国 `+66`
- 实时能力：后台端使用 WebSocket 推送，不做高频轮询
- 架构方式：模块化单体优先，后续按增长拆分
- 二期预留：收银 App 与离线同步能力

这套方案比较适合当前点餐系统场景，兼顾开发效率、可维护性、事务稳定性和实时性能。
